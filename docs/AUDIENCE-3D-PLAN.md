## 0. 한 줄 요약
카드/지하철/버스/멤버십 **오디언스 데이터**를 3축(교차) 지형으로 **파고들며 탐색**하는 기능. 매체 성과 3D(`components/Media3DExplorer.tsx`)의 UX 패턴(질문형 뷰·읽는 법 배지·드릴다운·항목 온오프·극단값 제외·전체창)을 **오디언스 도메인에 이식**한다. **백엔드 무변경** 전제.

## 1. 배경 / 왜 보류인가
- 매체 성과 3D는 "광고 성과(노출·전환·CPA)"를 다룬다. 오디언스 3D는 "**누가(연령·성별·지역) 어디에(업종·역·정류장) 몰리나**"를 다루는 **성격이 다른 분석**이다.
- 따라서 **성과·인사이트(매체 성과) 화면에 넣는 것은 부적절**하다. 오디언스 탐색 흐름(2단계: 오디언스 탐색)에 속한다.
- 규모가 큰 신규 작업이라 우선순위 보류. 본 문서로 기획만 확정해 둔다.

## 2. 배치 / 진입점 (결정 필요)
오디언스 3D는 **성과분석이 아니라 오디언스 탐색**에 속한다. 후보:
- **(A) 각 탐색 화면 내 시각화 토글** — 카드/지하철/버스/멤버십 화면에 `2D | 3D 탐색` 토글 추가(매체 성과의 `현재/정제 2D/3D 탐색` 토글과 동형). → 데이터 도메인과 화면이 일치, 가장 자연스러움. **권장.**
- (B) 사이드바 신규 메뉴 "오디언스 3D 탐색"(오디언스 탐색 그룹) — 단일 진입점에서 데이터소스를 탭으로 전환. → 진입점은 명확하나 화면 중복.
- 사이드바 구조는 `components/DmpSidebar.tsx`의 `TABS`/`GROUP_ORDER` 참고(현재 그룹: home/define/explore/extract/insight/system).

> 권장: **(A)** — 카드/지하철/버스/멤버십 각 화면에 3D 탐색 토글을 추가하고, 공통 `Audience3DExplorer` 컴포넌트를 데이터 어댑터와 함께 재사용.

## 3. 분석 뷰 정의 (질문형)
매체 3D와 동일하게 **각 뷰 = 하나의 질문 + 읽는 법 배지 + 드릴다운**.

| 데이터소스 | 뷰(질문) | 축 (X · Y · Z) | 좋은/의미 있는 위치 | 드릴다운(셀·막대 클릭) |
|---|---|---|---|---|
| 카드 | **소비 지형** — 어느 지역의 어느 연령이 많나? | 지역(시도) · 연령대 · 이용자수(높이) | 특정 지역×연령 봉우리 = 집중 소비층 | 그 셀(지역+연령)의 **업종 TOP·성별 분포** |
| 카드 | **업종 지도** — 어느 연령이 어느 업종을 쓰나? | 업종(대분류) · 연령대 · 이용자수 | 업종×연령 봉우리 = 핵심 소비 조합 | 그 조합의 **지역 분포·금액대** |
| 지하철 | **통근 지형** — 어느 역이 어느 연령대에 붐비나? | 역(TOP N) · 연령대 · 이용량 | 역×연령 봉우리 = 타깃 승하차 밀집 | 그 역의 **시간대 분포(출퇴근)** |
| 버스 | **정류장 지형** — 정류장×연령×이용량 | 정류장 · 연령대 · 이용량 | 동일 | 그 정류장 상세 |
| 멤버십 | **가맹점 지형** — 가맹점×연령×이용량 | 가맹점 · 연령대 · 이용량 | 가맹점×연령 봉우리 | 그 가맹점 상세(적립앱·금액대) |

- 축은 **2개의 범주형(예: 지역·연령) + 1개의 수치(규모)** 조합 = ECharts-GL `bar3D`(category × category × value)로 자연스럽게 표현.
- 산점(scatter3D)은 오디언스에선 의미가 약함 → **bar3D(지형)** 를 기본으로.

## 4. 데이터 조달 방법 (백엔드 무변경 · 핵심)
현행 오디언스 API는 **1D 집계 + 일부 2-way(연령×성별)** 만 반환하고 **지역×연령·역×연령 같은 교차표는 미제공**한다. 따라서 **필터를 바꿔가며 다회 호출해 교차표를 프론트에서 조립**한다. (매체 3D '시간 지형'이 매체별 `/api/media?view=daily`를 병렬 조달하는 것과 동일 패턴.)

### 4.1 카드 소비 지형 (지역 × 연령)
```
for 시도 s in SIDO_LIST(17):   // 병렬(Promise.all)
  GET /api/dashboard?sido=s     // 응답의 ageChart({a,M,F,U}) 추출
matrix[s][age] = M+F+U          // 17 × 6(연령) 매트릭스 → bar3D 데이터 [xi, yi, z]
```
- 참고: `/api/dashboard`는 `sido/sex/age`를 **필터**로 받음(`app/api/dashboard/route.ts`). 시도별 1콜씩 → 17콜.
- 응답 필드: `ageChart`(연령×성별), `region`, `industry` 등 — `components/Dashboard.tsx`의 사용부 참고.

### 4.2 카드 업종 지도 (업종 × 연령)
- 업종(대분류) 필터를 돌려가며 호출하거나, 단일 응답의 `industry`가 연령 분해를 제공하지 않으면 **업종별 필터 호출**로 조립. (대분류 수 = 콜 수)

### 4.3 지하철/버스 통근 지형 (역 × 연령)
```
stations = (첫 호출) /api/transit → station_top(TOP N)
for st in stations:            // 병렬
  POST /api/transit { station_id: st }   // 응답 age_gender 추출
matrix[st][age] = M+F                     // N × 6 → bar3D
```
- `/api/transit`는 `p_station_id` 단건 필터(`app/api/transit/route.ts`) → 역별 1콜.
- 드릴다운(역 시간대): 그 역 응답의 `hour_dist` 사용(추가 콜 불필요).

### 4.4 멤버십 가맹점 지형 (가맹점 × 연령)
- `/api/membership`의 `partner_cd` 필터를 가맹점별로 호출 → age_gender 조립.

### 4.5 조달 비용/성능 (주의)
- 콜 수 = 축1 카디널리티(시도 17 / 역 TOP N / 가맹점 TOP N). **TOP N 제한**(예 12~20)으로 콜 수·렌더 부담 관리.
- **로딩 UX**: "N개 셀 조달 중…" 프로그레스. `keepPreviousData`·병렬·부분 실패 허용(빈 셀은 0).
- **캐시**: 동일 필터 조합 SWR 캐시 재사용. 첫 조달만 비용.
- **콜드 쿼리**: 폐루프류(1~2분)와 달리 오디언스 큐브 RPC는 빠름(수십 ms) — 다회 호출도 감내 가능. 단 동시 17~20콜은 서버 부하 고려해 동시성 상한(예 6) 적용 권장.

## 5. 인터랙션 모델 (매체 3D 재사용)
`components/Media3DExplorer.tsx`의 검증된 패턴을 그대로 이식:
- **좌상단**: 데이터소스 아이콘 + 뷰 제목 + **질문**(예: "어느 지역의 어느 연령이 많나?").
- **읽는 법 배지**: `X=지역 · Y=연령대 · 높이=이용자수 · 💡봉우리=집중 소비층`.
- **탐색 탭**: 뷰 전환(소비 지형 / 업종 지도 / …).
- **드릴다운**: 막대(셀) 클릭 → 브레드크럼(← 전체 · 소비 지형 › 서울×30대) + 그 셀의 세부(업종 TOP·시간대 등) 2D 차트로 파고들기.
- **우측 분석 패널**(숨김/보임): 항목 온/오프 미니팝업(예: 표시 지역/역 선택) + 선택 셀 상세 + **AI 인사이트**(상황·인사이트·제안 3줄, `/api/ai-recommend` 재사용).
- **극단값 제외 토글**(IQR 1.5×) + **기간/월 필터** + **전체창(유튜브식)**.

## 6. 렌더링 / 기술 표준
- **ECharts-GL** `bar3D`(범주×범주×값) 기본. 드릴 상세는 2D ECharts(막대/라인).
- **색은 고정 hex '브랜드 격리 다크' 도크트린**(`Media3DExplorer.tsx`의 `C` 팔레트) — 캔버스는 CSS `var()` 미해석. 라벨·범례는 밝은 hex.
- 헤드리스(Playwright)에서 WebGL은 `--use-gl=angle --use-angle=swiftshader`로 렌더 확인. three.js CDN(unpkg) 의존 모듈은 헤드리스 미표시(실브라우저 정상) — 오디언스 3D는 ECharts-GL만 쓰므로 해당 없음.

## 7. 컴포넌트 설계(제안)
```
components/
  Audience3DExplorer.tsx     // 공통 셸 (탭·읽는법·드릴다운·패널·AI) — Media3DExplorer에서 도메인 분리 리팩터
  audience3d/
    adapters.ts              // 데이터소스별 조달·매트릭스 조립 (card/transit/membership)
    views.ts                 // 뷰 정의(질문·축·guide·드릴 스펙)
```
- 이상적으로는 `Media3DExplorer`를 **도메인 무관 셸 + 어댑터**로 리팩터해 매체/오디언스가 공유. (당장은 복사 후 분리도 허용 — additive.)

## 8. 구현 단계 (착수 시)
1. **카드 소비 지형(지역×연령)** 1개 뷰 + 조달 어댑터 + 로딩 UX → 카드 화면에 `3D 탐색` 토글로 붙여 배포.
2. 카드 업종 지도 추가.
3. 지하철/버스 통근 지형(역×연령) — transit 어댑터.
4. 멤버십 가맹점 지형.
5. 공통 셸 리팩터(선택).

## 9. 리스크 / 결정 필요
- **배치**: §2의 (A) 화면 내 토글 vs (B) 신규 메뉴 — **PO 결정 필요**.
- **교차표 부재 항목**: 지역×연령 이상으로 3차 실측 축(예: 지역×연령×업종 동시)은 다회 호출 곱연산이 커짐 → 3D는 "범주×범주×규모"로 한정, 3번째 실측 차원은 **드릴다운**에서 해소(권장).
- **동시성/부하**: TOP N 제한 + 동시 호출 상한. 대량 조달이 문제면 백엔드 교차표 RPC 신설이 정공법(그 경우 백엔드 변경 필요 — 별도 승인).
- **모바일/폭 좁은 화면**: 3D + 우측 패널은 데스크톱 우선. 좁은 폭은 패널 숨김 기본.
